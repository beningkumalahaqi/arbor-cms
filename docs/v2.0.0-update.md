# v2.0.0 — Widget System, Form Types & Container Widgets

**Date:** March 8, 2026

---

## Summary

This is a major release that adds a complete Widget System to Arbor CMS. Pages now support dynamic widget areas where admins can compose layouts by adding, configuring, and reordering widgets through the admin UI. The release includes 12 built-in widgets across 5 categories, a container widget system for nesting (sections and columns), a form widget with a dedicated FormType model for persistent form definitions, and an admin interface for viewing form submissions.

---

## Changes

### 1. Widget System Architecture

**New files:**
- `lib/widgets/types.ts` — Core type definitions (WidgetDefinition, WidgetInstance, WidgetPropDefinition, WidgetAreaDefinition, etc.)
- `lib/widgets/registry.ts` — Widget registry with register/get/validate/build-defaults API
- `lib/widgets/index.ts` — Public re-exports for all widget types and functions
- `lib/widgets/renderer.tsx` — WidgetItem (single widget) and WidgetArea (area renderer) components
- `lib/widgets/renderers/registry.ts` — Renderer registry mapping widget types to React components
- `lib/widgets/definitions/` — 12 widget definition files
- `lib/widgets/renderers/` — 12 widget renderer files

**How it works:**
- Each widget type is defined in a definition file declaring its `type`, `label`, `description`, `icon`, `category`, and `propSchema`
- Definitions are registered in `lib/widgets/registry.ts`
- Each widget has a corresponding renderer component registered in `lib/widgets/renderers/registry.ts`
- Page templates declare `WidgetAreaDefinition[]` to specify where widgets can be placed
- The catch-all route loads widgets from the database and passes them to the template
- The admin page editor has a "Widgets" tab for managing widgets per area

### 2. Database Schema — Widget Model

**File:** `prisma/schema.prisma`

**New model — `Widget`:**
- `id` (cuid), `pageId` (relation to Page), `area` (string), `type` (string), `props` (JSON string), `sortOrder` (int)
- `parentId` (nullable self-relation for nesting), `slot` (string, default "")
- Self-relation `WidgetTree` for parent/children hierarchy (container widget support)
- Cascade delete: widgets are deleted when their parent page is deleted
- Indexed on: `pageId`, `pageId+area`, `parentId`

### 5. Container Widgets & Nesting

**Section** and **Columns** are container widgets that wrap other widgets instead of having their own content.

- Container widgets declare `isContainer: true` and `slots: WidgetSlotDefinition[]`
- Each slot defines a named area where child widgets can be placed
- Section has one slot: `"content"`
- Columns has up to three slots: `"column1"`, `"column2"`, `"column3"` (column3 only shown for 3-column layouts)
- Child widgets are stored with `parentId` and `slot` fields in the database
- Containers cannot be nested inside other containers (prevents infinite recursion)
- Container renderers use a render-prop pattern (`renderWidget` callback) to recursively render children without circular imports

### 6. Form Widget & FormType System

**New model — `FormType`:**
- `id` (cuid), `name` (string), `elements` (JSON array as string), `createdAt`, `updatedAt`
- Stores form structure independently from widgets — survives widget deletion
- Indexed on: `name`

**New field on `FormSubmission`:**
- `formTypeId` (string, default "") — links submissions to a FormType
- Indexed on: `formTypeId`

**How forms work:**
- A Form widget references a `formTypeId` in its props
- When a form is submitted, the submission is linked to the FormType
- FormTypes are auto-created when a form widget is first saved (or on first submission)
- When creating a new form widget, admins can choose to reuse an existing FormType or create a new one
- If a linked FormType is deleted, the widget editor shows an error banner with an option to create a new one
- Form elements support: text input, email input, textarea, select, checkbox, radio buttons

**New API routes:**
- `GET /api/form-types` — List all form types
- `POST /api/form-types` — Create a new form type
- `DELETE /api/form-types?id=xxx` — Delete a form type

### 7. Widget API Routes

**New route — `/api/widgets`:**
- `GET ?pageId=xxx` — Get all widgets for a page
- `POST` — Create a new widget (validates type, props, parent container)

**New route — `/api/widgets/[id]`:**
- `GET` — Get a single widget
- `PUT` — Update widget props
- `DELETE` — Delete widget and cascade-delete children

**New route — `/api/widgets/reorder`:**
- `POST` — Reorder widgets by ID array

**New route — `/api/widgets/areas`:**
- `GET ?pageType=xxx` — Get widget area definitions for a page type

**New route — `/api/widgets/form-submit`:**
- `POST` — Public form submission endpoint (no auth, auto-creates FormType if needed)

**New route — `/api/widgets/page-list`:**
- `GET ?path=xxx` — Get child pages for the Page List widget

### 8. Admin UI — Widget Editor

**File:** `components/admin/widget-editor.tsx`

The page editor (`/admin/pages/[id]`) now has a "Widgets" tab alongside the "Content" tab:
- Widgets are organized by area (defined by the page template)
- Each area shows its widgets with inline editing panels
- Add Widget dialog groups available widgets by category with icons and descriptions
- Inline editing: click a widget to expand its property form
- Reorder: up/down arrows to move widgets within an area
- Delete: trash icon with confirmation
- Container widgets show nested slot management when editing
- Form widgets show a reuse dialog (pick existing FormType or create new)
- Widgets with missing FormTypes show an error banner with "Create Form Type" action

**File:** `components/admin/form-elements-editor.tsx`

A specialized editor for configuring form fields:
- Add/remove form elements (text, email, textarea, select, checkbox, radio)
- Configure labels, names, placeholders, required flag, options
- Reorder elements via up/down buttons

### 9. Admin UI — Form Submissions

**New pages:**
- `/admin/forms` — Lists all form submission groups with Form Name, Form Type, count, and last submission date
- `/admin/forms/[id]` — Shows individual submissions in a table with dynamic columns based on form fields
- Supports bulk delete and individual submission deletion
- Accessible from the admin sidebar

### 10. Widget Rendering Pipeline

**Public site rendering flow:**
1. Catch-all route (`app/[[...slug]]/page.tsx`) loads page + widgets from database
2. Widgets are parsed with `parentId` and `slot` fields
3. Page template receives widgets and renders `<WidgetArea>` components
4. `WidgetArea` filters for top-level widgets (no parent) in its area
5. `WidgetItem` renders each widget via a switch statement
6. Container widgets (section, columns) receive `allWidgets` + `renderWidget` callback
7. Container renderers filter children by `parentId` + `slot` and call `renderWidget` recursively

**Admin preview:**
- The live preview in the page editor also renders widgets using the same rendering pipeline
- The preview updates in real-time as widgets are added, edited, or reordered
