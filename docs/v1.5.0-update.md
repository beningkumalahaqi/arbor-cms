# v1.5.0 — Site Navigation & Footer

**Date:** March 6, 2026

---

## Summary

This update adds a responsive navigation bar and a configurable footer to the live (public) site. Both are managed through the admin Settings page with per-page control over navigation visibility. A new `SiteSettings` database model stores global configuration, and the `Page` model gains fields for navigation control.

---

## Changes

### 1. Database Schema Updates

**Files:** `prisma/schema.prisma`, `prisma/migrations/20260306120000_add_nav_footer_settings/migration.sql`

**New model — `SiteSettings`:**
- `navigationEnabled` (int, default 1) — toggle the nav bar on/off
- `navigationLogo` (string) — path to logo image in storage
- `navigationTitle` (string, default "Arbor CMS") — site title displayed in nav
- `footerEnabled` (int, default 1) — toggle the footer on/off
- `footerLogo` (string) — path to logo image in storage
- `footerText` (string) — text displayed in the footer
- Singleton pattern — only one row exists, upserted on save

**New fields on `Page`:**
- `showInNav` (int, default 0) — whether the page appears in the navigation bar
- `navLabel` (string, default "") — custom label for the nav link; if empty, defaults to Title Case of the slug

### 2. Site Navigation Component

**File:** `components/site/site-navigation.tsx`

A responsive, sticky navigation bar for the live site:
- **Desktop:** horizontal nav links with logo and site title on the left
- **Mobile:** hamburger button opens a side panel overlay from the right with backdrop
- Active page link highlighted with accent background
- Only **top-level published pages** with `showInNav` enabled appear
- Child pages (nested under other pages) are excluded from navigation
- Navigation label defaults to Title Case of the page slug (e.g., `about-us` → `About Us`)

### 3. Site Footer Component

**File:** `components/site/site-footer.tsx`

A standard footer rendered on all public pages:
- Displays optional logo image and configurable text
- Automatic copyright year
- Responsive layout — stacks vertically on mobile, side-by-side on desktop

### 4. Site Layout Wrapper

**File:** `components/site/site-layout.tsx`

A wrapper component that combines navigation + page content + footer:
- Conditionally renders nav and footer based on their enabled/disabled state
- Uses `min-h-screen` flexbox layout to push footer to the bottom
- The catch-all route fetches all layout data server-side via Prisma in parallel

### 5. Page Editor — Navigation Fields

**File:** `app/admin/pages/[id]/page.tsx`

For **top-level pages** (pages without a parent), the editor now shows a "Navigation" section:
- **Show in Navigation** — checkbox toggle
- **Navigation Label** — text input (only shown when checkbox is on), with placeholder showing the default Title Case label
- These fields are saved alongside content and status via the existing PUT endpoint

Child pages do not show these fields since only top-level pages can appear in navigation.

### 6. Settings Page — Navigation & Footer Sections

**File:** `app/admin/settings/page.tsx`

Two new settings cards added below the existing Theme card:

**Navigation settings:**
- Enable/disable toggle for the navigation bar
- Site title input (displayed next to the logo)
- Logo image selector (uses the existing ImageField + ImageSelectorModal from file manager)

**Footer settings:**
- Enable/disable toggle for the footer
- Footer text input
- Logo image selector

Settings auto-save on change — toggle and image changes save immediately, text inputs save on blur.

### 7. API Routes

**New — `GET /api/site-settings`:**
- Returns current site settings (navigation + footer config)
- No authentication required for GET
- Returns defaults if no settings exist yet

**New — `PUT /api/site-settings`:**
- Upserts site settings (creates if none exist, updates otherwise)
- Authentication required
- Accepts partial updates — only provided fields are changed

**New — `GET /api/site-navigation`:**
- Public endpoint (no auth required)
- Returns navigation items (top-level published pages with `showInNav` enabled) and footer configuration
- Navigation labels default to Title Case of the slug if `navLabel` is empty

**Updated — `PUT /api/pages/[id]`:**
- Now accepts `showInNav` (boolean) and `navLabel` (string) fields
- Values are persisted to the Page model alongside content and status

### 8. Catch-All Route Integration

**File:** `app/[[...slug]]/page.tsx`

The catch-all route now:
- Fetches page data, site settings, and nav pages in **parallel** via `Promise.all` for performance
- Wraps page template output in `SiteLayout` component
- Passes navigation and footer configuration as props
- No client-side API calls — all data resolved server-side
