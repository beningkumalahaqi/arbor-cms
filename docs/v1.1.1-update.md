# v1.1.1 — Page Template System

**Date:** March 5, 2026

---

## Summary

This update introduces a **Page Template** system that connects each page type to a dedicated rendering component. Previously, the public catch-all route used a single hardcoded layout for all page types. Now, every page type must have exactly one registered template that controls how its content renders on the public site.

---

## Changes

### 1. Page Template Types

**File:** `lib/page-template/types.ts`

Defines the core interface and type for templates:

- **`PageTemplateProps`** — the props every template receives: `content` (the page's content fields), `pageType` (the page type name), and `fullPath` (the URL path).
- **`PageTemplateComponent`** — a function component type that accepts `PageTemplateProps` and returns JSX.

### 2. Page Template Registry

**File:** `lib/page-template/registry.ts`

A registry that maps page type names to their template components, following the same pattern as the page-types registry:

- **`register(pageTypeName, template)`** — registers a template for a page type. Throws if a template is already registered for that type.
- **`getTemplate(pageTypeName)`** — retrieves the template component for a given page type.
- **`getAllTemplates()`** — returns all registered template mappings.

Built-in registrations:
- `"home"` → `HomeTemplate`
- `"content"` → `ContentTemplate`

### 3. Built-in Templates

**File:** `lib/page-template/home-template.tsx`

The Home template renders a centered hero-style layout — large title and description, vertically centered on the page. Designed to visually distinguish the home page from standard content pages.

**File:** `lib/page-template/content-template.tsx`

The Content template renders a standard article layout — constrained-width container with title and prose-formatted description. This matches the previous rendering behavior for all page types.

### 4. Public Route — Template-Based Rendering

**File:** `app/[[...slug]]/page.tsx`

The catch-all route no longer contains hardcoded HTML. Instead, it:

1. Looks up the page from the database by `fullPath`.
2. Calls `getTemplate(page.pageType)` to find the registered template.
3. Returns a 404 if no template is registered for the page type.
4. Calls the template function with `content`, `pageType`, and `fullPath`.

### 5. Barrel Exports

**File:** `lib/page-template/index.ts`

Exports `PageTemplateProps`, `PageTemplateComponent`, `getTemplate`, and `getAllTemplates` for use across the application.

### 6. Updated Guide

**File:** `guide/how-to-create-a-new-page-type.md`

The guide now includes two additional steps:

- **Step 3:** Create a page template component in `lib/page-template/`.
- **Step 4:** Register the template in `lib/page-template/registry.ts`.

## How to Add a Template for a New Page Type

1. Create a component file in `lib/page-template/` (e.g., `blog-post-template.tsx`).
2. Export a function that accepts `PageTemplateProps` and returns JSX.
3. Import and register it in `lib/page-template/registry.ts`:
   ```typescript
   import { BlogPostTemplate } from "./blog-post-template";
   register("blog-post", BlogPostTemplate);
   ```

See `guide/how-to-create-a-new-page-type.md` for the full step-by-step.

## Behavior Summary

| Rule | Enforcement |
|------|------------|
| Each page type has one template | Registry throws on duplicate registration |
| Public pages render via templates | Catch-all route calls `getTemplate()` |
| Missing template returns 404 | `notFound()` called when no template found |
| Templates are server components | Kept in `lib/page-template/`, separate from page-type definitions |
| Templates receive structured props | `content`, `pageType`, `fullPath` |

---

## Impact

- No database schema changes.
- No migration required.
- Public page rendering now uses registered templates instead of a hardcoded layout.
- The Home page now renders with a distinct hero-style template.
- Any future page type must register a template to be renderable on the public site.
