# v2.2.0 — Environment Sync API Token Management Update

**Date:** March 24, 2026

---

## Summary

This update refines the existing API-based Environment Sync implementation by moving transport token management from environment variables into the target environment itself. Tokens are now generated in admin settings, stored in `SiteSettings`, and validated server-side from the database for all sync transport endpoints.

---

## Changes

### 1. DB-Managed Token Validation for Sync Transport

**Files:**
- `app/api/environment-sync/auth.ts`
- `app/api/environment-sync/pull/route.ts`
- `app/api/environment-sync/push/route.ts`
- `app/api/environment-sync/status/route.ts`

Transport auth now validates bearer tokens against `SiteSettings.environmentSyncToken` in the target environment database.

### 2. Target Environment Token Generation

**Files:**
- `app/api/environment-sync/token/route.ts`
- `app/admin/settings/environment-sync/page.tsx`
- `app/api/site-settings/route.ts`
- `prisma/schema.prisma`

The target environment now provides secure token generation/regeneration from `/admin/settings/environment-sync`. Generated tokens are stored in `SiteSettings.environmentSyncToken`. The settings API only exposes a boolean (`environmentSyncTokenConfigured`) and never returns the stored token value.

### 3. API-Based Sync Flow (Push/Pull)

Environment sync transport continues to use:

- `Authorization: Bearer <token>` header
- `GET /api/environment-sync/pull` — returns local sync payload
- `POST /api/environment-sync/push` — applies incoming payload locally
- `GET /api/environment-sync/status` — bearer token check for remote connectivity, session check for admin UI

Invalid or missing token returns `401 Unauthorized`.

### 4. Setup Instructions

1. In the **target environment**, go to `/admin/settings/environment-sync` and click **Generate Token**.
2. Copy the generated token immediately (shown once after generation).
3. In the **source environment**, set:
   - **Target Environment API URL**
   - **Target Environment API Token** (the copied target token)
4. Save settings and confirm connection in `/admin/environment-sync`.

### 5. Push / Pull Flow

- **Sync to Target Environment**
  1. Source builds payload
  2. Source calls target `POST /api/environment-sync/push` with bearer token
  3. Target validates token from DB and applies payload

- **Sync from Target Environment**
  1. Source calls target `GET /api/environment-sync/pull` with bearer token
  2. Target validates token from DB and returns payload
  3. Source applies payload locally

---

## Migration

Database migration required:

- Add `environmentSyncToken` column to `SiteSettings` (default empty string).

Runtime migration:

1. Ensure environment sync transport no longer relies on environment-variable tokens.
2. Generate token from target environment settings.
3. Update source environment target token setting with generated value.
