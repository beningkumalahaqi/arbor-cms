# v2.2.3 - Environment Sync Widget Page Mapping Fix

**Date:** March 25, 2026

---

## Summary

This release fixes a widget synchronization bug in Environment Sync where widgets could be skipped during apply when source and target environments had different page IDs for the same page paths.

The fix is intentionally minimal and keeps the existing sync architecture and ordering.

---

## Bug

Widgets were not consistently synced even though pages were present and synced.

Observed behavior:

- Page records synced successfully
- Some widgets were skipped with page-missing logic
- Widget counts could be lower than expected after sync

---

## Root Cause

Widget upsert validation checked whether `widget.pageId` existed in the target database.

In sync flows where pages were matched by `fullPath` (not by ID), the target could keep its own existing page ID while the incoming widget payload still referenced the source page ID.

As a result, widget sync treated valid widgets as orphaned and skipped them because `widget.pageId` did not exist locally, even though the corresponding page existed under the same `fullPath`.

---

## Fix Applied

File updated:

- `lib/sync/service.ts`

Change made:

- Added page ID resolution for widget upserts:
  - Build a map of payload page IDs to `fullPath`
  - Build a map of target `fullPath` to target page IDs
  - If `widget.pageId` does not exist locally, resolve it via page `fullPath`
  - Upsert widget using the resolved target page ID

What did not change:

- Existing sync payload structure
- Sync endpoint contracts (`/pull`, `/push`)
- Entity sync order
- Widget parent-child ordering logic
- Upsert-by-ID strategy for widgets

---

## Behavior Before vs After

Before:

- Widgets referencing source page IDs were skipped when target page IDs differed
- Sync could report success while widget data was incomplete

After:

- Widgets correctly attach to the matching target page by path when IDs differ
- Widget sync remains idempotent and uses existing upsert semantics
- Parent-child widget relationships continue to be preserved

---

## Developer Notes

- This is a compatibility fix for cross-environment ID drift while preserving existing architecture.
- The fallback only applies when widget page ID lookup fails; direct ID matches continue to behave exactly as before.
