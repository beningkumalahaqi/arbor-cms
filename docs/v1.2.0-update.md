# v1.2.0 — Page Tree, Icon System & Page Type Settings

**Date:** March 5, 2026

---

## Summary

This update introduces a modern collapsible page tree view, a curated icon system for page types, and configurable page type settings (icon and allowed children). The Home page is now auto-created during bootstrap, and the API enforces allowed-children restrictions when creating new pages.

---

## Changes

### 1. New Database Model — PageTypeSettings

**File:** `prisma/schema.prisma`

Added a new `PageTypeSettings` model to store admin-configurable settings per page type:

- `pageTypeName` (unique) — references a registered page type name
- `icon` (string, default `"file"`) — the display icon name from the curated set
- `allowedChildren` (string, default `"[]"`) — JSON array of page type names allowed as children

**Migration:** `20260305065826_add_page_type_settings`

### 2. Icon System

**File:** `lib/icons.ts`

A curated set of 20 SVG icons (24x24 viewBox) available for page types. Each icon has a `name`, `label`, and `path` (SVG path data). Includes common icons: file, home, document, folder, globe, layout, book, image, star, settings, mail, users, shopping-bag, calendar, tag, message, grid, list, link, and info.

Helper functions:
- `getIconByName(name)` — look up an icon by name
- `getDefaultIcon()` — returns the "file" icon as fallback

**File:** `components/ui/page-type-icon.tsx`

A `PageTypeIcon` component that renders an icon from its SVG path data. Accepts `iconPath`, optional `className`, and `size` props.

### 3. Page Type Settings API

**File:** `app/api/page-types/settings/route.ts`

New REST endpoint for managing page type settings:

- **GET** — Returns all `PageTypeSettings` records. Requires authentication.
- **PUT** — Upserts settings for a single page type. Validates: page type must be registered, icon must be in the curated set, `allowedChildren` must be an array of valid page type names. Requires authentication.

### 4. Page Type Settings Admin UI

**File:** `app/admin/page-types/page.tsx`

Redesigned page types admin screen with:

- **Icon picker** — Grid of selectable icons for each page type. Selected icon is highlighted.
- **Allowed children editor** — Toggle buttons for each registered page type. When no children are selected, all types are allowed (no restriction).
- **Edit/Save flow** — Click "Edit Settings" to enter edit mode per page type, save or cancel.
- Page types are displayed as cards with their current icon, description, properties list, and allowed children badges.

### 5. Collapsible Page Tree View

**File:** `components/admin/page-tree.tsx`

A modern tree view component replacing the previous flat table:

- **Hierarchical display** — Pages are nested under their parents with indentation.
- **Expand/collapse** — Nodes with children have toggle arrows. All nodes start expanded.
- **Icon display** — Each node shows its page type's configured icon.
- **Badges** — Page type and status badges (published = green, draft = amber).
- **Edit link** — Hover to reveal an "Edit" link for each page.
- **Empty state** — Shows a dashed-border placeholder when no pages exist.
- **Header row** — Column labels for "Page" and "Type / Status".

**File:** `app/admin/pages/page.tsx`

Updated to fetch pages with `?tree=true` which returns both pages and settings in a single request, then passes them to the `PageTree` component.

### 6. Auto-Create Home Page on Bootstrap

**File:** `app/api/bootstrap/route.ts`

After creating the first SuperAdmin user, the bootstrap endpoint now:

1. Checks if a home page already exists.
2. If not, creates a published home page at `/` with default content from the home page type definition.
3. Creates default `PageTypeSettings` for the home type with icon `"home"` and allowed children `["content"]`.

This ensures the page tree always has a root Home page after initial setup.

### 7. Allowed Children Enforcement in Page Creation

**File:** `app/api/pages/route.ts`

The `POST` handler now enforces allowed-children restrictions:

- When creating a page with a `parentId`, the API looks up the parent page's `PageTypeSettings`.
- If the parent has a non-empty `allowedChildren` list and the new page's type is not in the list, the request is rejected with a `400` error.
- The `?tree=true` query parameter on GET returns pages with their children and all settings in one response.

### 8. New Page Form — Allowed Children Filtering

**File:** `app/admin/pages/new/page.tsx`

The new page form now respects allowed-children settings:

- Fetches both pages and page type settings on load.
- When a parent page is selected, the page type dropdown is filtered to only show types allowed by the parent's settings.
- If the parent has no restrictions (empty `allowedChildren`), all types are shown.
- If the currently selected page type becomes unavailable after changing parents, the selection is reset.

### 9. Updated Admin Sidebar

**File:** `components/admin/admin-shell.tsx`

Added "Page Types" as a navigation item in the admin sidebar, linking to `/admin/page-types`.

### 10. Documentation Updates

- **`.github/copilot-instructions.md`** — Added PageTypeSettings model, icon system, page tree, allowed children, and page type settings documentation. Updated folder structure, API routes, extensibility, and CMS principles sections.
- **`README.md`** — Added Page Type Settings and Icon System sections. Updated Page Tree, Admin Panel descriptions, project structure, and setup instructions.
- **`guide/how-to-create-a-new-page-type.md`** — Added Step 5 for configuring icon and allowed children in the admin UI.

## Behavior Summary

| Feature | Behavior |
|---------|----------|
| Home page auto-creation | Created on bootstrap with icon "home" and allowed children ["content"] |
| Icon selection | Admin UI picker from curated set, stored in PageTypeSettings |
| Allowed children | Configurable per page type, enforced server-side on page creation |
| Empty allowed children | Means no restriction — all page types allowed |
| Page tree view | Collapsible hierarchy with icons, badges, and edit links |
| Tree API | `GET /api/pages?tree=true` returns pages + settings |

---

## Impact

- Requires running `npx prisma migrate dev` to apply the new migration.
- Existing deployments without a Home page should run bootstrap again or create one manually.
- No breaking changes to existing page creation — allowed children only restrict when settings are configured.
